# metaverse-chainlab-v2

## 1. Scope

This skill governs V2 work for the Metaverse Blockchain Experiment Platform.
V2 starts after V1 completion.
V2 extends V1 into a modular, multi-source, dual-chain/cross-chain replay experiment platform.
V2 does not implement production blockchain deployment or production cross-chain bridges.

## 2. Required Starting Checks

```powershell

git log --oneline -12
```

```text
Do not push.
```

## 5. V2 / V3 Boundary

```text
V2 may implement local virtual-time dual-chain and cross-chain replay.
V2 may implement baseline cross-chain protocol models.
V2 may add public-chain imported trace skeletons.
V2 must not implement production multi-server Fabric.
V2 must not claim production cross-chain bridge capability.
V2 must not implement full MetaFlow unless a later explicit stage says so.
V3 is reserved for production-like deployment, multi-server networks, and full protocol experiments.
```

## 7. Data Truth Labels

```text
synthetic_replay:
  Generated workload + executor replay. Not real chain execution.

chain_backed_replay:
  Replay using trace generated by Fabric smoke. Trace source is real Fabric test-network, current run is replay.

fabric_smoke:
  CLI/WSL Fabric test-network run that deploys chaincode and invokes transactions.

existing_trace:
  User-provided or previously generated trace.

public_chain_imported_trace:
  Imported public-chain data. Externally real, but semantic access-set / delta / commutativity may be unknown.

production_deployment:
  V3 planned only.
```

## 8. Planned vs Runnable Rule

```text
Any config with runnable: false or status: planned must never be executed.
It can be displayed in UI as planned.
It can be validated as a known future topology.
It must not be passed to executor replay as runnable.
```

## 9. Default Validation Commands

```powershell
$env:PYTHONPATH = (Get-Location).Path
python -m pytest tests -q
python -m pytest backend/tests -q
python scripts/v0_sanity.py

cd executor
go test ./...
cd ..

cd frontend
npm.cmd run build
cd ..

git diff --check
git status --short
```

If a stage only changes docs, at minimum run `git diff --check` and `git status --short`.
If config files are added, also run Python tests that scan configs.
If Go or frontend is touched, run Go tests or frontend build respectively.
