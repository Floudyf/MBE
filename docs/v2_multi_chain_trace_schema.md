# V2.4 Multi-chain Trace Schema

## Goal

V2.4 defines the input schema for future multi-chain and cross-chain replay stages. It adds a cross-chain stage-record JSONL schema, a multi-chain trace metadata schema, a small readable sample, and a validator.

V2.4 only defines schema and validation. Schema validation is not replay execution, and the sample trace is not an experiment result.

## Cross-chain Trace Record

Each JSONL line is one stage record for one cross-chain transaction. Required fields are:

```text
schema_version
cross_tx_id
stage_id
stage
source_chain
target_chain
chain_id
submit_time_ms
status
data_truth_label
```

Recommended fields include:

```text
scenario
protocol
tx_id
asset_id
user_id
amount
commit_time_ms
finality_time_ms
timeout_deadline_ms
stage_latency_ms
access_list
read_set
write_set
metadata
```

Stage values:

```text
created
source_lock
source_finality
cert_generated
target_mint
target_finality
completed
timeout
refunded
failed
```

Status values:

```text
created
submitted
committed
finalized
completed
timeout
refunded
failed
```

Data truth labels:

```text
synthetic_replay
existing_trace_replay
fabric_chain_backed_trace_replay
public_chain_imported_trace_semantic_unknown
planned_cross_chain_replay
```

## Multi-chain Trace Metadata

The metadata file describes the whole trace file: format, compression, truth label, chain list, declared protocols, record counts, declared future metrics, and limitations.

Each chain declaration includes:

```text
chain_id
role
backend
block_interval_ms
finality_depth
```

Supported chain roles:

```text
source
target
intermediate
observer
```

Metadata may declare planned metrics, but V2.4 does not compute cross-chain metrics and does not claim a replay result.

## Sample Trace

The sample files are:

```text
trace/samples/v2_cross_trace_small.jsonl
trace/samples/v2_multi_chain_trace_meta.json
```

The sample contains two cross transactions: one completed path and one timeout/refund path. It uses `synthetic_replay` as a schema sample label. It is not generated by a dual-chain replay engine.

## Validator

`trace/validator/cross_chain_trace_validator.py` validates records, JSONL files, metadata files, and trace/meta consistency. It reads JSONL line by line and checks required fields, stage/status/truth-label enums, non-negative timestamps, time ordering, chain declarations, and record counts.

## Public-chain Imported Trace Rule

The schema allows `public_chain_imported_trace_semantic_unknown`. That label does not imply reliable access lists, read/write sets, commutative update semantics, update type, or delta semantics. The validator does not require those optional fields for semantic-unknown records.

## Non-goals

V2.4 does not run dual-chain replay. V2.4 does not run cross-chain protocols. V2.4 does not implement MetaFlow, committee bridge, or Pending Pool. V2.4 does not start Docker, Fabric, or `network.sh`. V2.4 does not connect to public-chain live nodes and does not require an archive node.

## Relationship To Later Stages

V2.5 may use this schema as input for a local virtual-time dual-chain replay engine. V2.6 may add cross-chain protocol baselines. Those later stages must still keep planned configs from being treated as runnable before their substrate exists.
