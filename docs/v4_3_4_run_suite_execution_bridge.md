# V4.3.4 Run Suite Execution Bridge

## A. Stage Position

V4.3.4 is the Run Suite Execution Bridge.

It is not a new runtime. It is not a complete batch scheduler. It is not run registry unification. The goal is only to let the Run Experiment page execute selected supported matrix rows through existing runners and return a small RunGroup result.

The flow is:

```text
Run Matrix Preview
-> Select Rows
-> Execute supported rows using existing runners
-> Return RunGroup preview/result
-> Show child run ids and artifact entry hints
```

## B. Supported Scope

This round supports real execution only for:

- `quick_validation`
- `v4_realism_validation`

`v4_realism_validation` continues to call the existing V4 realism runner behind `/api/v4/realism/smoke`.

`quick_validation` may call the most stable existing V3 quick validation path when low risk. If reusing the backend service cleanly would require touching old route logic or reconstructing hidden Composer payloads, the bridge may return a preview-only child result with a clear warning.

This round keeps these suite types as preview / dry-run:

- `main_experiment`
- `comparison_experiment`
- `ablation_experiment`
- `workload_sensitivity`
- `topology_scaling`

Reason: these formal, comparison, and ablation experiments require deeper integration with the existing `FormalMetatrackExperimentPanel` payload. This round must not force-build a synthetic payload and accidentally change formal runner semantics.

## C. Execution Model

The execution bridge uses four concepts:

- `RunMatrixRow`: a preview row generated by experiment-flow.
- `SelectedRunRow`: a user-selected row submitted for dry-run or execution.
- `RunGroup`: one lightweight batch request from the Run Experiment page.
- `ChildRun`: one row result from an existing runner or a preview-only decision.

`RunGroup` does not replace the historical run registry. It only returns child run id, status, summary, artifact hints, warnings, and blocked reasons.

## D. Boundaries

- V4 realism still uses `/api/v4/realism/smoke` behavior and the existing `v4_realism_runner`.
- Quick validation uses existing runner semantics or remains preview-only when clean reuse is not low risk.
- No Go runtime changes.
- No `executor/` changes.
- No formal runner semantic changes.
- No run registry unification.
- No large real-data batch execution.
- Planned workloads remain blocked.
- Production and non-claim fields remain false.
