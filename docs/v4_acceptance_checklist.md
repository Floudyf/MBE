# V4 Acceptance Checklist

## 1. Purpose

This checklist prevents V4 from becoming another preview/MVP layer that is mislabeled as real runtime. A V4 capability is accepted only when the evidence comes from actual runtime behavior.

## 2. Round 1: Docs And Skill Reset

Accepted when:

- README status mentions V4 planning and preserves V3-final baseline.
- `AGENTS.md` no longer describes the project as V0-only.
- V3 skill contains a frozen-baseline notice.
- V4 skill exists.
- V4 roadmap, architecture, stage docs, gap plan, migration plan, and checklist exist.
- No backend/frontend/Go runtime code is changed.
- `git diff --check` passes.

Truth label:

```text
v4_planning_only
```

## 3. Round 2: V4.0 Real Node Foundation

Accepted when:

- Implemented/verified: `mbe-node` starts with node id and shard id in `once` mode and has a truthful `server` skeleton.
- Implemented/verified: `mbe-client` generates signed transaction JSONL. It does not implement RPC submit in V4.0.
- Implemented/verified: signed transaction model includes sender, receiver, nonce, value, state keys, payload, timestamp, signature, tx id, public key metadata, and source kind.
- Implemented/verified: invalid signatures are rejected.
- Implemented/verified: duplicate tx ids are rejected.
- Implemented/verified: stale and future nonce violations are rejected by policy.
- Implemented/verified: every node owns an independent mempool.
- Implemented/verified: mempool capacity/TTL/wait metrics are represented; priority remains deterministic FIFO in V4.0.
- Implemented/verified: mempool logs are generated by admission behavior.
- Implemented/verified: V3 remains runnable under `go test ./...`.

```text
node_admission_log.csv
```


## 4. Round 3: V4.1 Network / Consensus / Commit

Accepted when:

- Implemented/verified: multiple V4.1 runtimes run with long-running server semantics and test-controlled graceful shutdown.
- Implemented/verified: each node listens on its own localhost TCP address.
- Implemented/verified: nodes exchange real TCP messages.
- Implemented/verified: TX_GOSSIP appears in real send/receive logs and receiving-node mempool admission.
- Implemented/verified: leader selects transactions from mempool.
- Implemented/verified: leader builds and broadcasts block proposals as PBFT_PRE_PREPARE.
- Implemented/verified: validators receive PrePrepare and send Prepare.
- Implemented/verified: validators collect Prepare quorum and send Commit.
- Implemented/verified: nodes commit after Commit quorum.
- Implemented/verified: basic ViewChange/NewView state transition is covered; production view-change proof, checkpoint, and stable log are not implemented.
- Implemented/verified: block commit logs are generated from real consensus state.
- Implemented/verified: V3 remains runnable under `go test ./...`.

## 5. Round 4: V4.2 State / Cross-shard / Recovery

Accepted when:

- committed blocks are executed deterministically;
- account/state transitions change persistent state;
- receipts are generated;
- state root is derived from actual state;
- block/state/receipt/tx-index storage is written;
- honest nodes in the same shard have matching block hash and state root;
- restart recovery reloads committed height and state root;
- cross-shard tx follows SourceLock / RelayCert / TargetCommit / SourceFinalize or Refund;
- fault injection affects real node/network behavior;
- frontend/backend can run or display V4 Realism Mode;
- V3 remains runnable.

V4.2 verification result:

- Implemented/verified: deterministic execution updates account state and receipts.
- Implemented/verified: persistent state db derives deterministic state roots from real state updates.
- Implemented/verified: durable block, receipt, and tx-index artifacts are written.
- Implemented/verified: honest-node consistency check reports matching committed height, block hash, receipt root, and state root.
- Implemented/verified: recovery reloads committed height and state root.
- Implemented/verified: cross-shard SourceLock / RelayCertificate / TargetCommit / SourceFinalize and Timeout / Refund / Abort paths are exercised.
- Implemented/verified: backend V4 realism smoke API and frontend Realism Mode are present.
- Not claimed: production PBFT, full Byzantine security, checkpoint/stable-log PBFT, Ethereum MPT compatibility, Fabric/EVM live backend, production atomic cross-shard commit, or measured superiority over BlockEmulator in every scenario.

## 6. BlockEmulator Surpass Checklist

MBE can claim it is moving beyond BlockEmulator-style realism only after it has evidence for:

- real signed transaction admission;
- independent per-node mempool;
- real local P2P send/receive;
- PBFT-style real message consensus;
- block commit through consensus;
- persistent block/state/receipt storage;
- state root consistency check;
- cross-shard state machine;
- recovery after node restart;
- network/node fault injection;
- BlockEmulator trace/config/result bridge;
- MetaTrack-aware routing comparison;
- metaverse workload support;
- frontend Realism Mode and reproducibility report.

Do not claim measured superiority until direct comparison experiments exist.

## 7. V4.3 BlockEmulator-surpass Realism Closure

Accepted when:

- signed transactions bind `sender` to `address(public_key)`;
- sender/public-key mismatch is rejected by `tx.Verify` and mempool admission;
- cross-shard relay certificate is sent over real TCP P2P;
- fault policy delay/drop is applied in the real P2P transport path;
- BlockEmulator-style CSV rows are converted to signed transaction JSONL and verified;
- backend and frontend expose V4.3 controls and artifacts;
- V3 runtime tests remain green.
