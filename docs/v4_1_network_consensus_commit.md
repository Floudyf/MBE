# V4.1 Network / Consensus / Commit

## 1. Stage Goal

V4.1 turns V4.0 real nodes and mempools into a real local multi-node block-commit runtime.

Target segment:

```text
per-node mempool -> real local P2P -> block proposer -> PBFT-style consensus -> committed block
```

## 2. Allowed Scope

V4.1 may add or modify:

```text
executor/realism/p2p/
executor/realism/router/
executor/realism/block/
executor/realism/consensus/pbft/
executor/realism/storage/
executor/realism/node/
executor/cmd/mbe-node/
executor/cmd/mbe-supervisor/
```

V4.1 may implement:

- long-running node server;
- TCP or gRPC local P2P loop;
- address table;
- peer manager;
- typed message codec;
- tx gossip;
- block proposal broadcast;
- shard leader block proposer;
- PBFT-style PrePrepare / Prepare / Commit;
- basic ViewChange / NewView;
- 2f+1 quorum;
- block database MVP;
- consensus certificate MVP;
- node-level network and consensus logs.

## 3. Forbidden Scope

V4.1 must not claim:

- production PBFT;
- complete Byzantine safety;
- complete checkpoint/stable log PBFT;
- persistent full state db;
- cross-shard commit;
- production-grade database durability;
- paper-final performance superiority.

## 4. PBFT-style Runtime Requirements

Each consensus node must maintain:

```text
view_id
sequence_id
leader_id
request_pool
prepare_votes
commit_votes
locked_block
committed_block
timeout
```

Required message types:

```text
PRE_PREPARE
PREPARE
COMMIT
VIEW_CHANGE
NEW_VIEW
```

Quorum checks:

```text
f = floor((n - 1) / 3)
prepare_quorum = 2f + 1
commit_quorum = 2f + 1
```

Votes must come from received node messages, not generated counts.

## 5. Block Requirements

A V4.1 block must include:

```text
block_height
previous_hash
tx_root
state_root_before
state_root_after_placeholder
receipt_root_placeholder
proposer_id
shard_id
timestamp
tx_list
signature
```

`state_root_after` and `receipt_root` may remain placeholders until V4.2, but the block hash, transaction root, proposer identity, and commit certificate must be real.


## 7. Acceptance Criteria

A V4.1 smoke run passes only if:

1. multiple nodes run as long-running processes;
2. each node listens on its own address;
3. nodes exchange real P2P messages;
4. tx gossip is observed from real send/receive logs;
5. leader selects transactions from the shard mempool;
6. leader broadcasts a block proposal;
7. validators receive PrePrepare and emit Prepare;
8. nodes collect Prepare messages and emit Commit after quorum;
9. nodes commit the block after Commit quorum;
10. a leader timeout can trigger basic ViewChange / NewView;
11. block commit logs are generated by the consensus path;
12. V3 remains runnable.

## 8. BlockEmulator Alignment

V4.1 is the stage that aligns with BlockEmulator's core emulator realism: nodes, transaction pool, block packaging, PBFT-style consensus, TCP/gRPC communication, and chain/block commit.

V4.1 must not copy BlockEmulator code. It should use BlockEmulator as a reference baseline for acceptance and comparison.

## 9. Stage Truth Label

```text
v4_real_p2p_consensus_commit
```

This label means real local node messaging, real block proposal, and PBFT-style real message commit are implemented. It does not imply production PBFT or full persistent state execution.
