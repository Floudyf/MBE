# V4.1 Network / Consensus / Commit

## 1. Stage Goal

V4.1 turns V4.0 real nodes and mempools into a real local multi-node block-commit runtime.

Target segment:

```text
per-node mempool -> real local P2P -> block proposer -> PBFT-style consensus -> committed block
```

## 2. Allowed Scope

V4.1 may add or modify:

```text
executor/realism/p2p/
executor/realism/router/
executor/realism/block/
executor/realism/consensus/pbft/
executor/realism/storage/
executor/realism/node/
executor/cmd/mbe-node/
executor/cmd/mbe-supervisor/
```

V4.1 may implement:

- Implemented/verified: long-running node server with short-duration smoke exit support.
- Implemented/verified: TCP local P2P loop using the Go standard library.
- Implemented/verified: address table and supervisor run-plan generation.
- Implemented/verified: peer manager with automatic local port support in tests.
- Implemented/verified: typed JSON message codec.
- Implemented/verified: TX_GOSSIP over real TCP send/receive.
- Implemented/verified: block proposal broadcast via PBFT_PRE_PREPARE.
- Implemented/verified: shard leader block proposer using the leader's mempool `PopReady`.
- Implemented/verified: PBFT-style PrePrepare / Prepare / Commit message flow.
- Implemented/verified: basic ViewChange / NewView state and logs.
- Implemented/verified: 2f+1 quorum checks.
- Implemented/verified: lightweight committed block log MVP.
- Implemented/verified: node-level network, PBFT, quorum, view-change, proposal, and commit logs.

## 3. Forbidden Scope

V4.1 must not claim:

- production PBFT;
- complete Byzantine safety;
- complete checkpoint/stable log PBFT;
- persistent full state db;
- cross-shard commit;
- production-grade database durability;
- paper-final performance superiority.

## 4. PBFT-style Runtime Requirements

Each consensus node must maintain:

```text
view_id
sequence_id
leader_id
request_pool
prepare_votes
commit_votes
locked_block
committed_block
timeout
```

Required message types:

```text
PRE_PREPARE
PREPARE
COMMIT
VIEW_CHANGE
NEW_VIEW
```

Quorum checks:

```text
f = floor((n - 1) / 3)
prepare_quorum = 2f + 1
commit_quorum = 2f + 1
```

Votes must come from received node messages, not generated counts.

## 5. Block Requirements

A V4.1 block must include:

```text
block_height
previous_hash
tx_root
state_root_before
state_root_after_placeholder
receipt_root_placeholder
proposer_id
shard_id
timestamp
tx_list
signature
```

`state_root_after` and `receipt_root` may remain placeholders until V4.2, but the block hash, transaction root, proposer identity, and commit certificate must be real.

## 7. Acceptance Criteria

A V4.1 smoke run passes only if:

1. Implemented/verified: multiple nodes run as long-running V4.1 runtimes in the Go integration smoke.
2. Implemented/verified: each node listens on its own localhost TCP address, with automatic port allocation in tests.
3. Implemented/verified: nodes exchange real TCP messages.
4. Implemented/verified: TX_GOSSIP appears in real send/receive logs and node mempool admission.
5. Implemented/verified: leader selects transactions from the shard mempool.
6. Implemented/verified: leader builds and broadcasts block proposals as PBFT_PRE_PREPARE.
7. Implemented/verified: validators receive PrePrepare and emit Prepare.
8. Implemented/verified: nodes collect Prepare messages and emit Commit after quorum.
9. Implemented/verified: nodes commit the block after Commit quorum.
10. Implemented/verified: basic ViewChange / NewView state transition is covered by PBFT tests; production proof, checkpoint, and stable log are not implemented.
11. Implemented/verified: block commit logs are generated by the consensus path.
12. Implemented/verified: V3 remains runnable under `go test ./...`.

## 8. BlockEmulator Alignment

V4.1 is the stage that aligns with BlockEmulator's core emulator realism: nodes, transaction pool, block packaging, PBFT-style consensus, TCP/gRPC communication, and chain/block commit.

V4.1 must not copy BlockEmulator code. It should use BlockEmulator as a reference baseline for acceptance and comparison.

## 9. Stage Truth Label

```text
v4_real_p2p_consensus_commit
```

This label means real local node messaging, real block proposal, and PBFT-style real message commit are implemented. It does not imply production PBFT or full persistent state execution.
